# Бизнес-правила AddressBook

Этот файл — единый источник подтверждённой продуктовой логики справочника. Перед планированием, уточняющими вопросами и изменением AddressBook его нужно читать вместе с `CODEX-RULES.MD` и `CODEX.MD`.

Если новое сообщение пользователя уточняет или отменяет правило из этого файла, приоритет имеет последнее уточнение. После реализации уточнение нужно сразу внести сюда, чтобы оно не потерялось между чатами.

## Состояние верхнего поиска

- Текст в поисковой строке до выбора конкретной подсказки — незавершённый поиск.
- Запрос подсказок выполняется с debounce 500 мс; устаревший запрос отменяется через `AbortSignal`.
- Нажатие Enter не переключает вкладку: пользователь остаётся в разделе, где начал поиск.
- После Enter значение сохраняется в `q` и становится глобальным фильтром всех таблиц и разделов.
- `q` и текст строки не сбрасываются при навигации между разделами.
- Переход между «Все контакты» и «Кадровая структура» с активным `q` повторяет поиск людей на выбранной вкладке.
- Поиск считается завершённым только после выбора конкретного человека или оргструктуры в выпадающем списке. Тогда `q` удаляется, а поисковая строка очищается.
- Крестик в заполненной строке очищает локальный текст, список подсказок и параметр `q`, сохраняя текущий раздел и остальные query-параметры.
- Выпадающий список делает один `multiSearch` с категориями `PERSONADDRESSBOOK` и `ORGSTRUCTURE`; отдельный запрос `groups` из поискового input запрещён.
- Категории единого ответа обрабатываются независимо: отсутствие данных одной категории не должно скрывать доступный контент другой.
- На вкладке «Кадровая структура» выпадающий список ищет только оргструктуру и не запрашивает категорию `PERSONADDRESSBOOK`.
- Ограничение dropdown не меняет Enter-сценарий: подтверждённый по Enter запрос людей на вкладке кадровой структуры остаётся глобальным.
- Результаты оргструктуры в dropdown отделяются подписью «Найдено в кадровой структуре».

## Поиск людей

- Люди загружаются через `multiSearch`, категория — `PERSONADDRESSBOOK`.
- `query` берётся из верхней поисковой строки.
- Поиск людей во «Всех контактах» и «Кадровой структуре» выполняется по всей компании.
- Если поиск людей запускается внутри выбранного подразделения, выбранная структура перестаёт ограничивать запрос: `orgFilter` сбрасывается и не передаётся.
- Enter показывает всех найденных людей и сохраняет текущую вкладку.
- Клик по человеку в подсказках завершает поиск, переключает на «Все контакты» и показывает только карточку выбранного человека.
- Карточка выбранного человека на маршруте «Все контакты» рендерится готовым `src/Components/Adressbook/Profile/Profile.tsx`; приложение передаёт базовые данные результата поиска и `pid`, а подробные данные загружает внутренний запрос Profile.
- При переходе к конкретному человеку глобальный `q` не должен восстанавливаться.
- Запрос и нормализация `multiSearch` используют подтверждённые поля `PERSONADDRESSBOOK.data.content`, `pbasic`, `jbasic`, `junit`, `jposition`, `personUuid` без вымышленных fallback-схем.

## Поиск и просмотр оргструктуры

- Поиск оргструктуры выполняется backend-поиском; dropdown получает и показывает плоский список результатов. Frontend не обходит дерево оргструктуры для формирования поисковых подсказок.
- Подтверждённый контракт `ORGSTRUCTURE.data.content[]`: обязательные `id` и `fullName`, дополнительные `typeName`, `layer`, `layerName`, `parents`.
- Клик по оргструктуре в подсказках завершает поиск, очищает строку, переводит пользователя на вкладку «Кадровая структура» и загружает сотрудников выбранного узла через `orgFilter=<id>`.
- Для выбранного узла отображается название оргструктуры, список её сотрудников и навигация «Наверх»/вложенные подразделения.
- При просмотре выбранного подразделения его строковый UUID передаётся в `multiSearch` как `orgFilter`.
- Запрос `groups` без `id` возвращает текущую структуру авторизованного пользователя, которая может находиться глубоко в иерархии; `null`/отсутствующий `id` не означает корень компании.
- При входе в кадровую структуру без `id` экран показывает возвращённое подразделение как полноценный экран узла, а его `id` используется как `orgFilter` для загрузки сотрудников.
- Открытие конкретного узла выполняет `groups?id=<UUID>`.
- В `id` нельзя передавать объект или строку `[object Object]`.
- Один пользовательский переход должен запускать не более одного запроса структуры; устаревшие запросы отменяются.
- Кнопка «Наверх» показывается только при наличии родительского узла и скрывается, когда `current.id === parentTree.id`.
- Когда `current.id === parentTree.id`, экран считается верхней точкой доступной иерархии и вместо таблицы текущего подразделения показывает карточки его дочерних структур.
- При поиске людей на вкладке «Кадровая структура» заголовок — «Результаты поиска», рядом отображается таблица людей, а слева — подразделения, в которых найдены сотрудники.
- **CRITICAL — требуется отдельное продуктовое обсуждение:** источник данных и точные правила фильтрации списка подразделений при поиске людей пока не согласованы. Нельзя подменять решение показом всей структуры или самостоятельно выводить окончательный контракт из данных первой страницы сотрудников.

## Недавние во «Всех контактах»

- При пустом `q` раздел «Все контакты» показывает «Недавние».
- История должна содержать только людей: запрос выполняется с `paths=persons`, а не `paths=all`.
- Курсы, сервисы, FAQ, оргструктуры и другие неперсональные контексты не должны попадать в список даже при смешанном ответе backend.
- Текущий размер выборки — 4 записи.
- «Недавние» отображаются той же таблицей `EmployeeTable → AdressBook → Table`, что и остальные списки сотрудников; упрощённый список текстовых записей истории запрещён.
- Выбор строки истории отмечается через `selectSearchHistory` и раскрывает `Profile` непосредственно в таблице без активации глобального `q` и без перехода на отдельный маршрут.

## Избранное

- «Избранное» является дочерним экраном раздела «Все контакты»: при переходе в избранное верхний таб «Все контакты» остаётся активным.
- Экран «Избранное» показывает таб «Все» и отдельный таб для каждой группы с `isCustom=true`; названия и порядок групп приходят с backend.
- В табе «Все» объединяются сотрудники всех загруженных custom-групп с дедупликацией по идентификатору человека; в табе группы показываются только её участники.
- Активная группа хранится в query-параметре `groupId`; переключение групп сохраняет глобальный `q` и остальные query-параметры.
- Звёздочка сотрудника по-прежнему управляет специальной custom-группой «Избранные», а не всеми пользовательскими группами.
- Списки «Избранное» и результаты поиска во «Всех контактах» рендерятся общей таблицей `src/Components/Adressbook` через единый адаптер приложения.
- Адаптер передаёт только фактически используемые таблицей и базовой частью Profile поля `AdressbookPerson`: идентификатор, ФИО по частям, фото, табельный номер, должность, блок, структуру, телефоны и данные отсутствия. Основной email остаётся в модели приложения только для действия «связаться»; почта Profile, адрес, мессенджер, часовой пояс, руководитель и оргпуть загружаются внутренним Profile API и не дублируются в `AdressbookPerson`.
- Сначала загружаются группы пользователя через `GET srv/v7/people/teams`.
- Для действия звёздочки нужная группа имеет `isCustom=true` и имя «Избранные» без учёта регистра и внешних пробелов.
- Участники каждой custom-группы загружаются через `GET srv/v7/people/teams/{id}?page=0&size=20&isCustom=true`.
- При каждом входе на экран «Избранное» группы и участники запрашиваются заново; долговременный promise-кэш между входами запрещён.
- Глобальный `q` применяется и к таблице избранного.
- Групповые действия применяются ко всем сотрудникам активного таба: в конкретном табе — к его участникам, в табе «Все» — к дедуплицированному объединению custom-групп.
- «Написать всем» открывает почтовый клиент через `mailto:` и передаёт доступные email в `bcc`.
- «Поставить встречу» скачивает пустой шаблон `.ics` с доступными email участников; тему и время пользователь заполняет в календарном приложении.
- Сотрудники без email пропускаются, а интерфейс сообщает их количество. Если доступных email нет, групповые действия недоступны.

## Прокрутка и пагинация

- Заголовок справочника, верхний поиск и навигационные вкладки находятся вне внутренней прокрутки.
- В отдельный вертикальный scroll-контейнер помещается только контент активного маршрута под вкладками; `position: sticky` для этого сценария не используется.
- Базовый размер страницы списков сотрудников — 20.
- Текущая страница хранится в query-параметре `page`; UI использует нумерацию с 1, API — с 0.
- Результаты поиска людей и сотрудники подразделения используют серверные метаданные `multiSearch`.
- Участники custom-групп запрашиваются постранично. При отсутствии page-метаданных конец определяется по короткой или пустой странице; повтор одинаковой страницы останавливает дозагрузку.
- Страница сбрасывается при изменении запроса, custom-группы или подразделения. Для четырёх записей «Недавних» пагинация не показывается.

## Справочные телефоны

- Переключение категории сохраняет глобальный `q` и остальные query-параметры.
- Глобальный `q` фильтрует справочные телефоны по названию службы, зоне ответственности и номеру телефона.

## Контракты HTTP

- Продуктовые API вызываются только через общие экземпляры `http` (`/api-web/`) и `profileHttp` (`/api-mobile/`).
- Общий `http.*` снимает ровно внешнее транспортное поле `data`. Generic описывает уже извлечённое внутреннее значение.
- Feature-код не должен повторно читать `response.data`, моделировать внешний envelope или рекурсивно искать `data`/`body`/`result`.
- Вложенные бизнес-поля `data` сохраняются. Например, внешний `{ data: { PERSONADDRESSBOOK: { data: ... } } }` превращается в `{ PERSONADDRESSBOOK: { data: ... } }`.
- Реальный API нельзя вызывать прямым `fetch`; локальный fetch-mock обслуживает только `/api/*`.
- Системные заголовки задаёт общий запросник и не дублируются в feature-коде.
- Граница выгрузки AddressBook состоит только из `src/Components/Adressbook/**` и `src/apps/AddressBook/**`.
- Продуктовые функции запросов, endpoint-пути, параметры, типы ответов и нормализация AddressBook должны находиться внутри этих двух каталогов и выгружаться вместе с модулем.
- Снаружи остаются только общие экземпляры HTTP-клиента `src/http-requests/http.ts` и host-эмуляция; размещать feature-specific API-файлы AddressBook рядом с общим клиентом запрещено.
- Если сценарий уже переведён на реальный API, продуктовый код всегда продолжает использовать этот API. Возврат к mock endpoint, mock fixture или условный fallback на моки запрещён.
- Локальная демонстрация может подменять ответы только во внешнем host-слое на транспортной границе; поставляемые каталоги не должны знать, что ответ был сэмулирован.

## Локальное mock-окружение

- Закрытый корпоративный контур не считается доступным условием локальной разработки.
- Обычный `npm run dev` подменяет реальные пути `/api-web/*` и `/api-mobile/*` исключительно в `src/host/**`.
- Продуктовые каталоги `src/apps/AddressBook/**` и `src/Components/Adressbook/**` продолжают использовать целевые URL и не содержат переключателей, fallback-ответов или зависимостей на host-fixtures.
- Host-моки возвращают wire-контракты целевого окружения, включая внешнее поле `data` для общего `HttpRequest` и прямых запросов импортированного компонента.
- Поиск, история, оргструктура, Profile и пользовательские группы используют согласованный набор вымышленных сотрудников; изменения custom-групп сохраняются в `localStorage`.
- Сценарий mock-ответов выбирается существующим ключом `addressbook-mock-scenario`: `success`, `loading`, `empty` или `error`.
- Неизвестный путь внутри `/api-web/*`, `/api-mobile/*` или `/api/*` в mock-режиме должен получать контролируемый `404` и не уходить в закрытый контур.
- Реальный proxy включается только явным `VITE_ADDRESSBOOK_USE_REAL_API=true`.

## Граница поставки и интеграция таблицы

- Файлы и diff, переданные из целевого проекта, сначала переносятся буквально, без адаптации под локальную оболочку, исправления типов, API-вызовов, импортов или стилей.
- Совместимость целевого кода с локальной оболочкой разбирается только отдельной следующей итерацией после полного переноса.
- При переносе копируются только `src/apps/AddressBook/**` и `src/Components/Adressbook/**`; вместе они образуют самодостаточный feature-модуль.
- Оба каталога могут ссылаться друг на друга. AddressBook-specific API, типы, endpoint-пути и преобразования должны находиться внутри этой пары каталогов.
- Переданный снимок `src/Components/Adressbook/**` является исходником коллеги: все его файлы сохраняются на месте, структура и внутренняя композиция не заменяются.
- Изменения в компоненте коллеги должны быть минимальными. Модели и сценарии `src/apps/AddressBook/**` адаптируются к публичному контракту таблицы на границе приложения.
- Нельзя удалять исходные wrapper-компоненты, hooks, assets, тесты или вспомогательные файлы только потому, что новая обвязка их временно не использует.

## Известные API и порядок подключения

- `multiSearch` — поиск людей и плоский поиск оргструктуры через категории `PERSONADDRESSBOOK`/`ORGSTRUCTURE`; подключён.
- `history` — недавние люди; подключён.
- `people/teams` — пользовательские и кастомные группы, включая избранное; подключён для чтения.
- `posts/api/v1/addressbook/groups` / `getGroupHierarchy` — оргструктура; текущая интеграция использует endpoint групп.
- Profile API `/api-mobile/*` — подробная карточка сотрудника; подключён через `profileHttp`.
- Новые API будут добавляться по мере передачи контрактов; нельзя угадывать неподтверждённые формы ответа.

## Что нельзя терять при следующих изменениях

- Не возвращать моки вместо уже подключённых реальных API.
- Не добавлять `orgFilter` к глобальному поиску людей.
- Не сбрасывать глобальный `q` при обычной навигации.
- Не оставлять `q` после выбора конкретной подсказки.
- Не использовать `paths=all` для недавних контактов.
- Не смешивать историю людей с общей историей корпоративного поиска.
