# CODEX-RULES

## Стек проекта

- React 18
- TypeScript
- Vite
- styled-components
- `@reach/router`
- локальные типизированные stubs для `@pulse/ui/*`

## Обязательные правила разработки

- В новом и редактируемом коде использовать только именованные импорты.
- Исключение: для деклараций styled-components v5 использовать default import `styled`; helpers (`css`, `keyframes`, `createGlobalStyle`, `useTheme`) импортировать именованно.
- Код в `src/apps/AddressBook/**` считается неизменяемым импортированным модулем: без отдельного явного запроса пользователя нельзя менять его содержимое, структуру каталогов, имена файлов, импорты и последовательность кода.
- Правило именованных импортов не является основанием переписывать уже импортированный неизменяемый модуль.
- Не использовать `any`.
- Не добавлять собственный backend.
- Реальные frontend API-интеграции добавлять только по явному запросу пользователя через общий `src/http-requests/http.ts`; несовместимости зависимостей запросника с песочницей изолировать в host-обвязке.
- Не добавлять зависимости без необходимости.
- Не менять функциональность за пределами текущей итерации.
- При неоднозначности требований останавливать работу и запрашивать уточнение.
- TypeScript должен оставаться в `strict`-режиме.

## Правила по стилям

- Основной способ описания стилей: объектная форма `styled-components`.
- Анимации оформлять только через `css` и `keyframes`.
- Для совместимости с целевым styled-components v5 объявлять элементы через `styled.div(...)`, `styled.button(...)` и аналогичные свойства intrinsic elements.
- Не использовать `import { styled } from 'styled-components'` и `styled('div')(...)` в новом или редактируемом коде обвязки.

## Правила внешней обвязки

- Любые несовместимости импортированного модуля с локальной песочницей устранять только вне `src/apps/AddressBook/**`.
- Host-обвязку размещать в `src/host/**`, локальные реализации Pulse — в `src/stubs/pulse/**`, общие host-константы и провайдеры — над модулем приложения.
- Реальные host-зависимости разрешено заменять минимальными типизированными адаптерами или no-op stubs, если это не меняет продуктовую логику модуля.
- Публичный контракт stub-компонента должен точно соответствовать используемому контракту оригинала, включая имена transient-пропсов с `$`, варианты компонентов и семантику HTML-элемента.
- Нельзя исправлять ошибки интеграции изменением импортов, пропсов или структуры исходного модуля.
- Если неизменяемый и неиспользуемый файл мешает компиляции, его разрешено исключить через `tsconfig` только после проверки отсутствия импортов; исключение обязательно документировать в `CODEX.md`.

## Правила theme и design tokens

- Единственный источник темы локальной песочницы — host theme, переданный через `ThemeProvider` в `src/main.tsx`.
- Тип `styled-components/DefaultTheme` расширять только во внешней декларации `src/host/styled.d.ts`; декларации внутри импортированного модуля не редактировать.
- Структура host theme должна повторять фактически используемый контракт Pulse, а не заставлять модуль переходить на локальную схему темы.
- Все design tokens находятся непосредственно в `theme.tokens.current.*`; промежуточного уровня `core` нет.
- Фоны: `background.default`, `background.field`.
- Акценты: `accent.primary`, `accent.secondary`, `accent.tertiary`, `accent.base`.
- Текст: `text.primary`, `text.secondary`, `text.tertiary`, `text.onColor`, `text.placeholder`, `text.error`.
- Иконки: `icon.primary`, `icon.secondary`, `icon.tertiary`, `icon.onColor`.
- Слои: `layer['01']`, `layer['02']`, `layer['03']`.
- Границы: `border.gentle`, `border.strong`, `border.interactive`.
- Опасные состояния: `danger['01']`, `danger['02']`.
- Служебные состояния: `support.success`, `support.attention`, `support.warning`, `support.info`.
- Если исходный модуль обращается к дополнительному token-path, нужно расширить тип и значение host theme; нельзя заменять обращение внутри модуля или добавлять там hardcode.
- Каждый `theme.typography.*` — объект с `fontFamily`, `fontWeight`, `fontSize`, `lineHeight`, `letterSpacing`; его можно передавать через spread в styled-объект.
- Доступные варианты типографики перечислены в `docs/THEME.md`; нельзя моделировать typography как строку или локальную таблицу размеров.
- Скругления находятся в `theme.radii`, не в `theme.radius`; локальная обвязка предоставляет `sm`, `md`, `lg`, `pill`.
- В theme также доступны `borderWidths`, `breakpoints`, `layout`, `layoutColumnWidth`, `mediaQueries`, `screenSize`, `space` и `zIndices`.
- `theme.space` помечен в Pulse как deprecated и не должен становиться основой нового layout-кода.
- Pulse-stubs должны использовать host tokens, typography и radii. Нельзя возвращать старую параллельную схему `theme.colors`, `theme.spacing`, `theme.radius` или `theme.tokens.current.core`.
- Полный локальный контракт и порядок его расширения описаны в `docs/THEME.md`.

## Правила aliases

- Alias для одной зависимости должен быть синхронно описан в `vite.config.ts` и `tsconfig.json`.
- Более специфичный alias `@pulse/ui/components/*` должен разрешаться в локальные Pulse-stubs до общего `@pulse/*`.
- Для host-зависимостей (`i18next`, `react-i18next`, breadcrumbs и аналогичных) использовать внешние адаптеры, не менять пути импорта в исходном модуле.
- `@hrplatform/utils` и `@pulse/ui/theme` в локальной среде разрешать через синхронные Vite/TypeScript aliases; исходный `src/http-requests/http.ts` не адаптировать под песочницу.

## Правила HTTP-запросов

- Реальные API вызывать через общий `http` из `src/http-requests/http.ts`, построенный на `HttpRequest('/api-web/', httpRequestOptions)`.
- Не создавать параллельный fetch-клиент и не перехватывать `window.fetch` для реального API.
- Сохранять системные заголовки `HRP-Device-Type` и `X-HRP-Device-Type`, которые задаёт общий запросник.
- Передавать `AbortSignal` в долгоживущие поисковые запросы и отменять устаревший запрос при смене query.
- Локальный fetch-mock разрешён только для PoC endpoints `/api/*` и не должен перехватывать `/api-web/*`.
- Контракт запросника и текущая интеграция поиска описаны в `docs/HTTP-REQUESTS.md`.

## Правила Pulse UI

- Источник публичных контрактов компонентов — полная выгрузка типизации `@pulse/ui`, зафиксированная в `docs/PULSE-UI.md`.
- Нельзя придумывать отсутствующие props для удобства локального stub: например, у `Text` нет `tone` и `weight`, у `Avatar` нет `$initials`.
- `Text.variant` обязателен; цвет и отступы передаются через styled-system `ColorProps` и `SpaceProps`.
- Инициалы `Avatar` передаются через `$type="initials"` и `$text`; доступны размеры `xs`, `s`, `m`, `l`, `xl`, `xxl`.
- `Button` по умолчанию имеет `$type="primary"` и `$size="m"`; поддерживаемые типы и размеры перечислены в `docs/PULSE-UI.md`.
- `Loader` использует props `isOnColor`, `size`, `wrapped`, `children` без transient-префикса.
- В предоставленной выгрузке существует компонент `Empty`, но отсутствует `EmptyState`. Текущий `@pulse/ui/components/EmptyState` является отдельно документированным host-adapter по продуктовому требованию и не считается подтверждённым контрактом DS.
- Stub расширяется только после сверки соответствующего `types.ts`; несовместимость нельзя скрывать добавлением произвольного prop.
- Не создавать stubs для неиспользуемых компонентов заранее. Если import-path не подтверждён выгрузкой (например, плоский `IconButton` или `Spinner`), такой файл не должен оставаться доступным через alias.
- Ключевые контракты и само разрешение imports проверяются compile-time файлом `src/host/pulseUiContractAudit.ts`.

## Правила маршрутизации

- Единственный механизм клиентской маршрутизации — `@reach/router`.
- Дерево экранов объявлять через `Router`, динамические сегменты — через `path` и route props.
- Общий layout, который использует location, размещать внутри `LocationProvider`, но выше `Router`, чтобы он не перемонтировался между экранами.
- Для чтения URL использовать `useLocation`, для программных переходов — `useNavigate`, для декларативного редиректа — `Redirect`.
- Все публичные URL и builders динамических URL хранить в `routes/routePaths.ts`.
- Нельзя напрямую вызывать `window.history.pushState`, `replaceState`, подписываться на `popstate` или создавать собственный router context.
- Нельзя вручную разбирать pathname регулярными выражениями для выбора React-экрана.
- Query-параметры читать из `location.search`; router должен сам сохранять раздельные `pathname`, `search` и `hash`.
- `@reach/router@1.3.4` несовместим с React 18 `StrictMode`: пока библиотека не заменена, локальную точку входа запускать без `StrictMode`.
- После изменений маршрутизации проверять root redirect, прямые ссылки, query, динамические employee/department routes и browser back.

## Правила выполнения итераций

- Итерации выполняются строго последовательно.
- После завершения каждой итерации работа останавливается до следующей задачи.
- После каждой завершённой задачи или итерации обязательно обновлять `ROADMAP.md`.
- После каждой итерации обязательно выполнять:
  - `npm run typecheck`
  - `npm run build`

## Scope-ограничения PoC

- Реализуется frontend-демонстрация; единственная разрешённая реальная интеграция — явно подключённый поиск сотрудников через общий проектный запросник.
- Мобильная версия в scope PoC не входит.
- Рабочие группы, SberAgile, ФОС и прочие исключённые разделы не добавляются, пока это явно не потребуется в отдельной итерации.
