# CODEX-RULES

## Стек проекта

- React 18
- TypeScript
- Vite
- styled-components
- `@reach/router`
- локальные типизированные stubs для `@pulse/ui/*`

## Обязательные правила разработки

- Перед планированием, уточняющими вопросами, изменением кода или запуском команд обязательно полностью читать `docs/BUSINESS-RULES.md` вместе с `CODEX-RULES.MD` и `CODEX.MD`.
- Каждое подтверждённое пользователем бизнес-правило, сценарий, исключение или исправление прежней трактовки сразу фиксировать в `docs/BUSINESS-RULES.md` и при необходимости в профильной документации. Нельзя оставлять такие знания только в истории чата или `ROADMAP.md`.
- Перед повторным вопросом пользователю сначала проверять `docs/BUSINESS-RULES.md`; уже зафиксированную информацию повторно не запрашивать.
- При конфликте бизнес-правил приоритет имеет последнее явное уточнение пользователя; документ нужно обновить так, чтобы устаревшая трактовка не оставалась действующей.
- В новом и редактируемом коде использовать только именованные импорты.
- Исключение: для деклараций styled-components v5 использовать default import `styled`; helpers (`css`, `keyframes`, `createGlobalStyle`, `useTheme`) импортировать именованно.
- Код в `src/apps/AddressBook/**` считается неизменяемым импортированным модулем: без отдельного явного запроса пользователя нельзя менять его содержимое, структуру каталогов, имена файлов, импорты и последовательность кода.
- Переданный пользователем снимок `src/Components/Adressbook/**` считается исходником коллеги и источником истины: все входящие в снимок файлы должны оставаться на месте, а изменения внутри них допускаются только минимальные и непосредственно необходимые для интеграции.
- Адаптация приложения к таблице выполняется преимущественно в `src/apps/AddressBook/**`: приложение преобразует свои API-модели в публичный контракт `src/Components/Adressbook/**`, не заменяя внутреннюю композицию `AdressBook → People → Table → Profile`.
- Правило именованных импортов не является основанием переписывать уже импортированный неизменяемый модуль.
- Не использовать `any`.
- Не добавлять собственный backend.
- Реальные frontend API-интеграции добавлять только по явному запросу пользователя через общие экземпляры из `src/http-requests/http.ts`; несовместимости зависимостей запросника с песочницей изолировать в host-обвязке.
- Граница поставки AddressBook состоит только из `src/Components/Adressbook/**` и `src/apps/AddressBook/**`. Все feature-specific API-функции, endpoint-пути, параметры, типы ответов и нормализация обязаны находиться внутри этих каталогов.
- Два поставляемых каталога должны быть самодостаточны как единый модуль: они могут импортировать друг друга, но не должны зависеть от AddressBook-specific файлов за своими границами. Снаружи разрешена только общая инфраструктура проекта, явно перечисленная в правилах.
- Из поставляемых каталогов разрешено импортировать снаружи только общую инфраструктуру проекта, включая `src/http-requests/http.ts`; создавать рядом с ней файлы `directorySearch.ts`, `groups.ts`, `profile.ts` и другие AddressBook-адаптеры запрещено.
- Перед завершением каждой задачи проверять, что AddressBook не импортирует feature-файлы за пределами двух поставляемых каталогов.
- Не добавлять зависимости без необходимости.
- Не менять функциональность за пределами текущей итерации.
- При неоднозначности требований останавливать работу и запрашивать уточнение.
- TypeScript должен оставаться в `strict`-режиме.

## Правила по стилям

- Основной способ описания стилей: объектная форма `styled-components`.
- Анимации оформлять только через `css` и `keyframes`.
- Для совместимости с целевым styled-components v5 объявлять элементы через `styled.div(...)`, `styled.button(...)` и аналогичные свойства intrinsic elements.
- Не использовать `import { styled } from 'styled-components'` и `styled('div')(...)` в новом или редактируемом коде обвязки.

## Правила внешней обвязки

- Любые несовместимости импортированного модуля с локальной песочницей устранять только вне `src/apps/AddressBook/**`.
- Host-обвязку размещать в `src/host/**`, локальные реализации Pulse — в `src/stubs/pulse/**`, общие host-константы и провайдеры — над модулем приложения.
- Реальные host-зависимости разрешено заменять минимальными типизированными адаптерами или no-op stubs, если это не меняет продуктовую логику модуля.
- Публичный контракт stub-компонента должен точно соответствовать используемому контракту оригинала, включая имена transient-пропсов с `$`, варианты компонентов и семантику HTML-элемента.
- Нельзя исправлять ошибки интеграции изменением импортов, пропсов или структуры исходного модуля.
- Если неизменяемый и неиспользуемый файл мешает компиляции, его разрешено исключить через `tsconfig` только после проверки отсутствия импортов; исключение обязательно документировать в `CODEX.md`.

## Правила theme и design tokens

- Единственный источник темы локальной песочницы — host theme, переданный через `ThemeProvider` в `src/main.tsx`.
- Тип `styled-components/DefaultTheme` расширять только во внешней декларации `src/host/styled.d.ts`; декларации внутри импортированного модуля не редактировать.
- Структура host theme должна повторять фактически используемый контракт Pulse, а не заставлять модуль переходить на локальную схему темы.
- Все design tokens находятся в `theme.tokens.current.core.*`; уровень `core` обязателен.
- Для импортированного `src/Components/Adressbook/**` дополнительно сохраняются существующие Pulse-ветки `tokens.current.colors.*` и `tokens.current.shadows.*`; их предоставляет host theme, исходные обращения не переписываются.
- Фоны: `core.background.default`, `core.background.field`.
- Акценты: `core.accent.primary`, `core.accent.secondary`, `core.accent.tertiary`, `core.accent.base`.
- Текст: `core.text.primary`, `core.text.secondary`, `core.text.tertiary`, `core.text.onColor`, `core.text.placeholder`, `core.text.error`.
- Иконки: `core.icon.primary`, `core.icon.secondary`, `core.icon.tertiary`, `core.icon.onColor`.
- Слои: `core.layer['01']`, `core.layer['02']`, `core.layer['03']`.
- Границы: `core.border.gentle`, `core.border.strong`, `core.border.interactive`.
- Опасные состояния: `core.danger['01']`, `core.danger['02']`.
- Служебные состояния: `core.support.success`, `core.support.attention`, `core.support.warning`, `core.support.info`.
- Если исходный модуль обращается к дополнительному token-path, нужно расширить тип и значение host theme; нельзя заменять обращение внутри модуля или добавлять там hardcode.
- Каждый `theme.typography.*` — объект с `fontFamily`, `fontWeight`, `fontSize`, `lineHeight`, `letterSpacing`; его можно передавать через spread в styled-объект.
- Доступные варианты типографики перечислены в `docs/THEME.md`; нельзя моделировать typography как строку или локальную таблицу размеров.
- Скругления находятся в `theme.radii`, не в `theme.radius`; локальная обвязка предоставляет `sm`, `md`, `lg`, `pill`.
- В theme также доступны `borderWidths`, `breakpoints`, `layout`, `layoutColumnWidth`, `mediaQueries`, `screenSize`, `space` и `zIndices`.
- `theme.space` помечен в Pulse как deprecated и не должен становиться основой нового layout-кода.
- Pulse-stubs должны использовать host tokens, typography и radii. Нельзя использовать сплющённые пути `theme.tokens.current.text` и аналоги либо возвращать параллельные схемы `theme.colors`, `theme.spacing`, `theme.radius`.
- Полный локальный контракт и порядок его расширения описаны в `docs/THEME.md`.

## Правила aliases

- Alias для одной зависимости должен быть синхронно описан в `vite.config.ts` и `tsconfig.json`.
- Более специфичный alias `@pulse/ui/components/*` должен разрешаться в локальные Pulse-stubs до общего `@pulse/*`.
- Для host-зависимостей (`i18next`, `react-i18next`, breadcrumbs и аналогичных) использовать внешние адаптеры, не менять пути импорта в исходном модуле.
- `@hrplatform/utils` и `@pulse/ui/theme` в локальной среде разрешать через синхронные Vite/TypeScript aliases; исходный `src/http-requests/http.ts` не адаптировать под песочницу.

## Правила HTTP-запросов

- API `/api-web/*` вызывать через общий `http`, а Profile API `/api-mobile/*` — через общий `profileHttp`; оба экземпляра объявлены только в `src/http-requests/http.ts` и используют единые `httpRequestOptions`.
- Общий `HttpRequest` снимает одну внешнюю транспортную оболочку `data`; feature-код типизирует и читает уже внутреннее значение. Повторно читать `response.data` для транспортной оболочки запрещено.
- Не создавать запросники в feature-коде, не вызывать прямой `fetch` и не перехватывать `window.fetch` для реального API.
- Сохранять системные заголовки `HRP-Device-Type` и `X-HRP-Device-Type`, которые задаёт общий запросник.
- Передавать `AbortSignal` в долгоживущие поисковые запросы и отменять устаревший запрос при смене query.
- Локальный fetch-mock внутри поставляемых каталогов разрешён только для исторических PoC endpoints `/api/*`; продуктовый код не должен перехватывать `/api-web/*` или `/api-mobile/*`.
- Внешняя host-песочница в `src/host/**` может подменять реальные `/api-web/*` и `/api-mobile/*` на транспортной границе, сохраняя исходные URL, методы и wire-контракты целевого окружения.
- Локальный `npm run dev` использует host-моки по умолчанию; обращение к реальному закрытому контуру включается только явным `VITE_ADDRESSBOOK_USE_REAL_API=true`.
- После подключения реального API продуктовый код не имеет права откатываться на mock API, mock fixtures или fallback-моки. Моки подставляются только внешней host-обвязкой/песочницей на транспортной границе и не меняют endpoint, клиент или ветвление продуктового кода.
- Контракт запросника и текущая интеграция поиска описаны в `docs/HTTP-REQUESTS.md`.

## Правила Pulse UI

- Источник истины публичных контрактов — неизменённые выгрузки `docs/pulse-ui-types/raw/types-context.part-1.md` и `types-context.part-2.md`.
- `docs/PULSE-UI.md` — краткая памятка, а не замена полной типизации. При расхождении приоритет имеют raw-файлы и сгенерированная из них точная секция.
- Перед добавлением или изменением Pulse-компонента/stub обязательно прочитать `docs/pulse-ui-types/README.md`, найти компонент через `INDEX.md` или экспорт через `EXPORTS.md`, затем прочитать только соответствующие файлы `generated/src/**` и их необходимые зависимости.
- Нельзя читать обе raw-выгрузки целиком по умолчанию, восстанавливать props по памяти или полагаться только на краткую памятку.
- `INDEX.md`, `EXPORTS.md`, `manifest.json` и `generated/**` обновлять командой `npm run docs:pulse-types`; вручную их не редактировать.
- Нельзя придумывать отсутствующие props для удобства локального stub: например, у `Text` нет `tone` и `weight`, у `Avatar` нет `$initials`.
- `Text.variant` обязателен; цвет и отступы передаются через styled-system `ColorProps` и `SpaceProps`.
- Инициалы `Avatar` передаются через `$type="initials"` и `$text`; доступны размеры `xs`, `s`, `m`, `l`, `xl`, `xxl`.
- `Button` по умолчанию имеет `$type="primary"` и `$size="m"`; поддерживаемые типы и размеры перечислены в `docs/PULSE-UI.md`.
- `Loader` использует props `isOnColor`, `size`, `wrapped`, `children` без transient-префикса.
- Для пустых состояний использовать только `@pulse/ui/components/Empty`; компонента `EmptyState` в DS нет, создавать для него alias или host-adapter запрещено.
- `Empty` требует `type` и `description`; допустимые `type`: `start`, `noResults`, `wait`, `create`, `noData`. Props `illustration` в контракте нет.
- Stub расширяется только после сверки соответствующего сгенерированного `types.ts.md` и импортируемых им DS-типов; несовместимость нельзя скрывать добавлением произвольного prop.
- Не создавать stubs для неиспользуемых компонентов заранее. Если import-path не подтверждён выгрузкой (например, плоский `IconButton` или `Spinner`), такой файл не должен оставаться доступным через alias.
- Ключевые контракты и само разрешение imports проверяются compile-time файлом `src/host/pulseUiContractAudit.ts`.

## Правила маршрутизации

- Единственный механизм клиентской маршрутизации — `@reach/router`.
- Дерево экранов объявлять через `Router`, динамические сегменты — через `path` и route props.
- Общий layout, который использует location, размещать внутри `LocationProvider`, но выше `Router`, чтобы он не перемонтировался между экранами.
- Для чтения URL использовать `useLocation`, для программных переходов — `useNavigate`, для декларативного редиректа — `Redirect`.
- Все публичные URL и builders динамических URL хранить в `routes/routePaths.ts`.
- При встраивании AddressBook во внешний Reach Router родительский маршрут обязан захватывать дочерние URL через `path="/addressbook/*"`.
- Внутренний Router AddressBook использует `basepath="/"`, потому что его routePaths уже содержат полный публичный host path; нельзя одновременно наследовать внешний basepath и повторно добавлять его к абсолютным путям.
- Нельзя напрямую вызывать `window.history.pushState`, `replaceState`, подписываться на `popstate` или создавать собственный router context.
- Нельзя вручную разбирать pathname регулярными выражениями для выбора React-экрана.
- Query-параметры читать из `location.search`; router должен сам сохранять раздельные `pathname`, `search` и `hash`.
- `@reach/router@1.3.4` несовместим с React 18 `StrictMode`: пока библиотека не заменена, локальную точку входа запускать без `StrictMode`.
- После изменений маршрутизации проверять root redirect, прямые ссылки, query, динамические employee/department routes и browser back.

## Правила выполнения итераций

- Итерации выполняются строго последовательно.
- После завершения каждой итерации работа останавливается до следующей задачи.
- После каждой завершённой задачи или итерации обязательно обновлять `ROADMAP.md`.
- После каждой итерации обязательно выполнять:
  - `npm run typecheck`
  - `npm run build`

## Правила тестирования

- Для unit/component-тестов использовать только Jest и React Testing Library; не добавлять другой test runner или библиотеку взаимодействий без отдельного запроса.
- Добавление тестов не разрешает менять продуктовую логику. Если тест выявил расхождение поведения с требованиями, нужно зафиксировать его отдельно и запросить продуктовую правку, а не подгонять реализацию под тест.
- Базовые `render`, `renderHook` и остальные React Testing Library helpers импортировать из `src/test-utils/test-utils`; глобальный стек providers не расширять feature-провайдерами Addressbook.
- Provider-зависимые тесты Addressbook явно оборачивать локальным `AddressbookTestProvider` из `src/Components/Adressbook/test-utils`; тесты самого provider могут использовать `renderHook` с явным wrapper.
- Внешние модули мокать через `jest.mock`; моки типизировать через `jest.MockedFunction`, `jest.Mocked` или явно типизированный `jest.Mock`. `any`, `@ts-ignore` и нетипизированные заглушки запрещены.
- В `beforeEach` сбрасывать реализации и очереди ответов изменяемых моков через `mockReset`, затем задавать исходный сценарий теста. Одного `clearMocks` недостаточно для изоляции реализаций.
- Подменённые globals, timers, `fetch`, `Response`, storage listeners и другие внешние состояния обязательно восстанавливать в `afterEach`/`afterAll`.
- Продуктовые API тестировать моками общего `http`/`profileHttp`; глобальный `fetch` разрешено подменять только в тестах исторического `/api/*` mock-транспорта и legacy-клиентов, которые действительно используют fetch.
- Проверять наблюдаемое поведение и вызовы публичных границ. Не менять production-код ради достижения строки coverage и не дублировать внутреннюю реализацию в ожиданиях.
- Feature-specific fixtures, builders и providers хранить внутри границы поставки AddressBook; общую тестовую theme и базовый render — в `src/test-utils`. Production-код не должен импортировать `test-utils`.

## Scope-ограничения PoC

- Реализуется frontend-демонстрация; разрешены явно подключённые поиск сотрудников, Profile API `/api-mobile`, список пользовательских групп `/api-web/srv/v7/people/teams` и кадровая структура `/api-web/posts/api/v1/addressbook/groups` через общие проектные запросники.
- Мобильная версия в scope PoC не входит.
- Рабочие группы, SberAgile, ФОС и прочие исключённые разделы не добавляются, пока это явно не потребуется в отдельной итерации.
